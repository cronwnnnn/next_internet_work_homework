# 项目代码讲解与伪代码

本文档用于辅助实验报告中的“设计/实现功能过程”部分，按项目文件和关键函数说明代码逻辑。项目没有从零实现 MQTT/CoAP 协议栈，而是基于成熟协议库完成物联网数据采集、传输、存储、展示和对比实验。

## 1. 项目结构

```text
src/iot_protocol_lab/
├── config.py          # 全局配置：端口、Topic、传感器类型、数据库路径
├── mqtt_broker.py     # 本地 MQTT Broker
├── mqtt_simulator.py  # MQTT 传感器模拟器
├── dashboard.py       # MQTT 订阅采集 + Web Dashboard
├── storage.py         # SQLite 数据库操作
├── coap_server.py     # CoAP 服务端
├── coap_client.py     # CoAP 客户端与 RTT 统计
└── __init__.py
```

## 2. config.py

### 2.1 文件作用

`config.py` 集中保存项目配置，避免在多个文件中重复写端口、Topic 和传感器参数。

### 2.2 主要内容

```text
设置 MQTT 地址：
    MQTT_HOST = 127.0.0.1
    MQTT_PORT = 1883
    MQTT_TOPIC_ROOT = iot/lab

设置 CoAP 地址：
    COAP_HOST = 127.0.0.1
    COAP_PORT = 5683

设置数据库路径：
    项目根目录 / data / iot_messages.db

定义传感器类型：
    temperature:
        单位 C
        正常范围 18.0 到 32.0
        告警阈值 >= 35.0
    humidity:
        单位 %
        正常范围 35.0 到 75.0
        告警阈值 >= 85.0
    light:
        单位 lux
        正常范围 120.0 到 800.0
        告警阈值 >= 950.0
    air_quality:
        单位 AQI
        正常范围 20.0 到 120.0
        告警阈值 >= 150.0
```

## 4. mqtt_simulator.py

### 4.1 文件作用

该文件实现 MQTT 传感器模拟器，用于模拟多个物联网设备定时上传温度、湿度、光照、空气质量等数据。

### 4.2 build_payload(device_id, sensor_type, qos)

该函数根据设备编号、传感器类型和 QoS 生成一条传感器数据。

伪代码：

```text
函数 build_payload(device_id, sensor_type, qos):
    从 SENSOR_TYPES 中读取该传感器的配置
    读取正常范围 low, high

    生成一个随机数:
        如果随机数小于 0.08:
            生成一个超过告警阈值的异常值
        否则:
            在正常范围内生成随机值

    构造 JSON 数据:
        device_id      = 设备编号
        sensor_type    = 传感器类型
        value          = 传感器数值
        unit           = 单位
        sent_at        = 当前时间戳，用于计算延迟
        timestamp      = ISO 格式时间
        qos            = MQTT QoS 等级

    返回 JSON 数据
```

### 4.3 run_simulator(host, port, devices, interval, qos)

该函数连接 MQTT Broker，并持续发布模拟传感器数据。

伪代码：

```text
函数 run_simulator(host, port, devices, interval, qos):
    创建 MQTT Client
    连接 MQTT Broker
    启动 MQTT 网络循环

    根据 devices 参数生成设备编号列表:
        device_01, device_02, ...

    获取所有传感器类型:
        temperature, humidity, light, air_quality

    循环执行:
        对每个设备:
            随机选择一个传感器类型
            调用 build_payload() 生成数据
            构造 Topic:
                iot/lab/{device_id}/{sensor_type}
            将 JSON 数据发布到该 Topic
            在终端输出发布内容

        等待 interval 秒

    如果用户按 Ctrl+C:
        停止 MQTT 网络循环
        断开连接
```

### 4.4 parse_args()

伪代码：

```text
函数 parse_args():
    读取命令行参数:
        --host       Broker 地址
        --port       Broker 端口
        --devices    模拟设备数量
        --interval   发送间隔
        --qos        MQTT QoS，允许 0、1、2

    返回参数对象
```

## 5. dashboard.py

### 5.1 文件作用

该文件是 MQTT 主系统的核心，包含两个部分：

- MQTT 订阅采集端：订阅传感器数据，计算延迟和告警状态，写入 SQLite。
- Web Dashboard：通过 HTTP API 提供统计数据，并展示监控页面。

### 5.2 is_alert(sensor_type, value)

该函数判断某条传感器数据是否超过告警阈值。

伪代码：

```text
函数 is_alert(sensor_type, value):
    根据 sensor_type 查询告警阈值

    如果该传感器类型存在阈值，并且 value >= 阈值:
        返回 True
    否则:
        返回 False
```

### 5.3 start_mqtt_collector(host, port)

该函数启动 MQTT 订阅采集逻辑。

伪代码：

```text
函数 start_mqtt_collector(host, port):
    初始化数据库
    创建 MQTT Client

    定义 on_connect 回调:
        输出连接状态
        订阅 iot/lab/+/+
        订阅 QoS 设置为 2，便于测试发布端 QoS 0/1/2

    定义 on_message 回调:
        记录 received_at 当前接收时间

        尝试处理消息:
            将 MQTT payload 从 bytes 解码为字符串
            将字符串解析为 JSON

            读取 sensor_type
            读取 value
            使用 received_at - sent_at 计算 latency_ms
            调用 is_alert() 判断是否告警

            构造数据库记录:
                received_at
                topic
                device_id
                sensor_type
                value
                unit
                qos
                latency_ms
                alert
                payload

            调用 insert_mqtt_message() 写入 SQLite

        如果解析失败:
            输出错误信息

    将回调函数绑定到 MQTT Client

    定义后台线程 worker:
        循环尝试连接 Broker
        如果连接成功:
            进入 MQTT loop_forever()
        如果 Broker 暂未启动:
            等待 2 秒后重试

    启动后台线程
    返回 MQTT Client
```

## 6. storage.py

该文件封装 SQLite 数据库操作，包括建表、插入消息、查询最新数据、查询最近消息和统计数据。

```text

    如果 mqtt_messages 表不存在:
        创建表，字段包括:
            id
            received_at
            topic
            device_id
            sensor_type
            value
            unit
            qos
            latency_ms
            alert
            payload

    关闭数据库连接
```

### 6.3 insert_mqtt_message(row, db_path)

```text
函数 insert_mqtt_message(row, db_path):
    调用 init_db() 确保数据库和表存在
    连接 SQLite 数据库

    插入一条 MQTT 消息记录:
        received_at

        device_id
        sensor_type
        value

        qos

        alert

    提交事务
    关闭数据库连接
```

### 6.4 latest_by_sensor(db_path)

该函数查询每个设备、每类传感器的最新一条数据。

伪代码：

```text
函数 latest_by_sensor(db_path):
    调用 init_db()
    连接 SQLite 数据库

    查询 mqtt_messages:
        按 device_id 和 sensor_type 分组
        找到每组最大的 id
        根据最大 id 取出完整消息
        按 device_id 和 sensor_type 排序

    将查询结果转为字典列表
    返回列表
```

### 6.5 recent_messages(limit, db_path)

伪代码：

```text
函数 recent_messages(limit, db_path):
    调用 init_db()
    连接 SQLite 数据库

    查询 mqtt_messages:
        按 id 倒序排列
        只取 limit 条

    将查询结果转为字典列表
    返回列表
```

### 6.6 summary_stats(db_path)

该函数计算 Dashboard 顶部统计卡片需要的数据。

伪代码：

```text
函数 summary_stats(db_path):
    调用 init_db()
    连接 SQLite 数据库

    执行聚合查询:
        COUNT(*)              -> total_messages
        AVG(latency_ms)       -> avg_latency_ms
        MIN(latency_ms)       -> min_latency_ms
        MAX(latency_ms)       -> max_latency_ms
        SUM(alert)            -> alert_count

    如果某些统计结果为空:
        转换为 0

    对延迟保留 3 位小数
    返回统计字典
```

## 7. coap_server.py

### 7.1 文件作用

该文件实现 CoAP 服务端，用于接收传感器数据上报。它主要用于和 MQTT 主系统做协议对比。

### 7.2 SensorResource.render_post(request)

该函数处理客户端对 `/sensor` 资源的 POST 请求。

伪代码：

```text
类 SensorResource:
    异步函数 render_post(request):
        记录 received_at 当前接收时间

        尝试解析请求:
            将 request.payload 解码为字符串
            将字符串解析为 JSON
            读取 sent_at
            计算 latency_ms = received_at - sent_at

            构造响应:
                status = ok
                device_id
                sensor_type
                latency_ms
                received_at

            输出接收日志
            返回 CoAP 2.04 Changed 响应

        如果解析失败:
            构造错误响应:
                status = error
                error = 错误信息
            返回 CoAP 4.00 Bad Request 响应
```

### 7.3 run_server(host, port)

伪代码：

```text
异步函数 run_server(host, port):
    创建 CoAP 资源站点 root
    将 /sensor 路径绑定到 SensorResource
    创建 CoAP Server Context，监听 host:port
    输出监听地址
    等待程序持续运行
```

### 7.4 main()

伪代码：

```text
函数 main():
    使用 asyncio 运行 run_server()
```

## 8. coap_client.py

### 8.1 文件作用

该文件实现 CoAP 客户端，定时发送传感器数据到 CoAP 服务端，并统计每次请求的 RTT。

### 8.2 send_one(protocol, uri, index)

该函数发送一次 CoAP POST 请求。

伪代码：

```text
异步函数 send_one(protocol, uri, index):
    随机选择传感器类型
    读取该传感器的正常范围
    在正常范围内生成随机数值

    构造 JSON payload:
        device_id
        sensor_type
        value
        unit
        sent_at

    构造 CoAP POST 请求:
        method = POST
        uri = coap://host:port/sensor
        payload = JSON 编码后的字节

    记录 start = 当前高精度时间
    发送请求并等待响应
    记录响应返回时间

    计算 rtt_ms
    输出响应状态码、RTT 和响应内容
    返回 rtt_ms
```

### 8.3 run_client(host, port, count, interval)

伪代码：

```text
异步函数 run_client(host, port, count, interval):
    构造 uri:
        coap://host:port/sensor

    创建 CoAP Client Context
    等待 0.1 秒，确保上下文初始化完成

    创建空列表 rtts

    循环 index 从 1 到 count:
        调用 send_one() 发送一次请求
        将返回的 rtt_ms 加入 rtts
        等待 interval 秒

    输出 RTT 统计:
        count
        average RTT
        minimum RTT
        maximum RTT
```

### 8.4 parse_args()

伪代码：

```text
函数 parse_args():
    读取命令行参数:
        --host
        --port
        --count
        --interval

    如果 count < 1:
        输出参数错误

        输出参数错误

    返回参数对象
```

## 9. scripts/run.ps1

### 9.1 文件作用

该脚本用于解决 Windows 环境下 uv 默认缓存目录可能无法访问的问题。脚本会将 `UV_CACHE_DIR` 设置到项目内部的 `.uv-cache` 目录，然后再执行 uv 命令。

### 9.2 脚本伪代码

```text
读取用户传入的命令行参数

设置错误处理模式为 Stop
计算项目根目录 ProjectRoot
设置环境变量:
    UV_CACHE_DIR = ProjectRoot/.uv-cache

如果用户没有传入参数:
    输出使用方法
    退出

进入项目根目录
执行:
    uv 用户传入的参数

执行结束后:
    返回原来的目录
```

## 11. 系统运行流程总伪代码

### 11.1 MQTT 主流程

```text
启动 MQTT Broker:
    监听 127.0.0.1:1883

启动 Dashboard:
    初始化 SQLite
    后台连接 MQTT Broker
    订阅 iot/lab/+/+
    启动 Web 服务

启动 Sensor Simulator:
    连接 MQTT Broker
    循环生成传感器数据
    发布到对应 Topic

Dashboard 收到消息:
    解析 JSON
    计算延迟
    判断是否告警
    写入 SQLite

浏览器访问 Dashboard:
    每秒请求 API
    展示统计信息、最新数据和最近消息
```

### 11.2 CoAP 对比实验流程

```text
启动 CoAP Server:
    监听 coap://127.0.0.1:5683/sensor

启动 CoAP Client:
    循环 count 次:
        生成传感器数据
        POST 到 /sensor
        等待响应
        统计 RTT

实验结束:
    输出平均 RTT、最小 RTT、最大 RTT
```
