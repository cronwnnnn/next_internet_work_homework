## 1. 选题简介

随着物联网设备数量增长，传感器、网关、云平台之间需要频繁交换小体量、实时性较强的数据。传统 HTTP 协议虽然通用，但在低带宽、低功耗、多设备接入等场景下会带来较高连接和报文开销。因此，物联网系统通常使用更轻量的应用层协议，其中 MQTT 和 CoAP 是较有代表性的两类协议。

本实验选择“MQTT/CoAP 物联网协议分析与实现”作为结课作业题目，设计并实现一个简化的物联网数据采集与监控系统。系统主线基于 MQTT 实现多设备传感器数据发布、Broker 转发、订阅端采集、SQLite 存储和 Web Dashboard 展示；辅线基于 CoAP 实现传感器数据 POST 上报，并对比两种协议在通信模型和延迟表现上的差异。

本实验的目标包括：

- 理解 MQTT 的发布/订阅模型、Topic 设计和 QoS 机制。
- 理解 CoAP 的轻量 REST 模型和基于 UDP 的请求/响应方式。
- 实现一个可运行的 MQTT 物联网数据采集系统。
- 实现一个简化的 CoAP 数据上报实验。
- 对 MQTT 与 CoAP 的适用场景、通信开销和实验延迟进行分析。

## 2. 协议基本原理

### 2.1 MQTT 协议原理

MQTT 是一种面向物联网场景的轻量级消息传输协议，运行在 TCP 之上。它采用发布/订阅模型，通信双方不直接连接，而是通过 MQTT Broker 进行消息转发。

MQTT 系统中主要包含三类角色：

- Publisher：消息发布者，例如传感器设备。
- Broker：消息代理服务器，负责接收、过滤和转发消息。
- Subscriber：消息订阅者，例如监控平台或数据采集服务。

发布者将消息发送到某个 Topic，订阅者订阅相应 Topic 后即可接收消息。本实验中使用的 Topic 格式为：

```text
iot/lab/{device_id}/{sensor_type}
```

例如：

```text
iot/lab/device_01/temperature
iot/lab/device_02/humidity
```

MQTT 支持三种 QoS 等级：

| QoS | 含义 | 特点 |
|---|---|---|
| QoS 0 | 至多一次 | 不确认，可能丢失，开销最低 |
| QoS 1 | 至少一次 | 有确认，可能重复，可靠性较高 |
| QoS 2 | 只有一次 | 四步握手，可靠性最高，开销也最高 |

需要注意的是，MQTT 实际投递 QoS 由发布端和订阅端共同决定，实际等级为二者较小值。因此本实验中 Dashboard 订阅 QoS 设置为 2，便于测试发布端 QoS 0、1、2 的不同效果。

### 2.2 CoAP 协议原理

CoAP 是 Constrained Application Protocol 的缩写，是一种面向受限设备和受限网络的轻量级应用层协议。它通常运行在 UDP 之上，接口形式类似 HTTP 的 REST 模型，支持 GET、POST、PUT、DELETE 等方法。

CoAP 的特点包括：

- 基于 UDP，协议开销较小。
- 采用资源 URI 组织数据，例如 `coap://127.0.0.1:5683/sensor`。
- 支持 Confirmable Message，用确认机制提升可靠性。
- 适合低功耗设备、局域网设备发现、简单状态上报等场景。

与 MQTT 不同，CoAP 默认是请求/响应模型，不依赖 Broker。设备通常直接向服务器发送请求，例如传感器向 `/sensor` 资源 POST 当前数据。

## 3. 系统设计

### 3.1 总体架构

本实验系统由 MQTT 主系统和 CoAP 对比实验两部分组成。

MQTT 主系统架构如下：

```text
Sensor Simulator
       |
       | MQTT Publish
       v
MQTT Broker
       |
       | MQTT Subscribe
       v
Dashboard Collector
       |
       | Write
       v
SQLite Database
       |
       | Query
       v
Web Dashboard
```

CoAP 对比实验架构如下：

```text
CoAP Client
    |
    | POST /sensor
    v
CoAP Server
    |
    | Response
    v
RTT Statistics
```

### 3.2 功能模块

本实验代码主要包含以下模块：

| 文件 | 功能 |
|---|---|
| `src/iot_protocol_lab/mqtt_broker.py` | 启动本地 MQTT Broker |
| `src/iot_protocol_lab/mqtt_simulator.py` | 模拟多个物联网传感器发布数据 |
| `src/iot_protocol_lab/dashboard.py` | 订阅 MQTT 消息、写入数据库、提供 Web Dashboard |
| `src/iot_protocol_lab/storage.py` | SQLite 初始化、插入和查询 |
| `src/iot_protocol_lab/coap_server.py` | CoAP 服务端，接收传感器 POST 上报 |
| `src/iot_protocol_lab/coap_client.py` | CoAP 客户端，发送数据并统计 RTT |
| `src/iot_protocol_lab/config.py` | 端口、Topic、传感器类型等配置 |
| `scripts/run.ps1` | Windows 下统一设置 uv 缓存目录并运行命令 |

### 3.3 数据格式设计

传感器模拟器生成 JSON 格式数据，示例如下：

```json
{
  "device_id": "device_01",
  "sensor_type": "temperature",
  "value": 26.8,
  "unit": "C",
  "sent_at": 1779370900.123,
  "timestamp": "2026-05-22T10:00:00+00:00",
  "qos": 1
}
```

字段说明：

| 字段 | 含义 |
|---|---|
| `device_id` | 设备编号 |
| `sensor_type` | 传感器类型 |
| `value` | 传感器数值 |
| `unit` | 单位 |
| `sent_at` | 发送时间戳，用于计算端到端延迟 |
| `timestamp` | ISO 格式时间 |
| `qos` | 发布端 MQTT QoS |

系统支持四类模拟传感器：

| 类型 | 单位 | 正常范围 | 告警阈值 |
|---|---|---|---|
| temperature | C | 18.0 - 32.0 | 大于等于 35.0 |
| humidity | % | 35.0 - 75.0 | 大于等于 85.0 |
| light | lux | 120.0 - 800.0 | 大于等于 950.0 |
| air_quality | AQI | 20.0 - 120.0 | 大于等于 150.0 |

```text

```

Broker 允许匿名连接，方便实验环境下的传感器模拟器和 Dashboard 订阅端连接。

### 4.3 MQTT 传感器模拟器实现

`mqtt_simulator.py` 用于模拟多个传感器设备。程序可通过命令行指定设备数量、发送间隔和 MQTT QoS 等级。

示例命令：

```powershell
.\scripts\run.ps1 run iot-mqtt-simulator --devices 4 --interval 1 --qos 1
```

模拟器会随机选择传感器类型并生成数据，同时按一定概率生成超过阈值的异常数据，用于验证 Dashboard 告警功能。

### 4.4 Dashboard 与数据存储实现

`dashboard.py` 同时承担两个功能：

- MQTT Subscriber：订阅 `iot/lab/+/+`，接收所有设备的传感器数据。
- Web Server：提供 Dashboard 页面和统计 API。

Dashboard 接收到消息后，会计算端到端延迟：

```text
latency_ms = (received_at - sent_at) * 1000
```

然后将消息写入 SQLite 数据库。Web 页面每秒请求一次 API，展示：

- 总消息数
- 平均延迟
- 最小延迟
- 最大延迟
- 告警次数
- 各设备最新状态
- 最近消息列表

启动命令：

```powershell
.\scripts\run.ps1 run iot-dashboard
```

浏览器访问：

```text
http://127.0.0.1:8000
```

### 4.5 CoAP 数据上报实现

CoAP 服务端监听：

```text
coap://127.0.0.1:5683/sensor
```

客户端向该资源发送 POST 请求，服务端收到数据后返回确认响应。客户端统计每次请求的 RTT，并输出平均值、最小值和最大值。

启动服务端：

```powershell

```

启动客户端：

```powershell
.\scripts\run.ps1 run iot-coap-client --count 30 --interval 0.5
```

### 5.2 MQTT QoS 对比实验

实验目的：比较不同 QoS 等级下 MQTT 消息投递的延迟和可靠性差异。

实验方法：

分别运行以下命令，每组运行相同时间，例如 60 秒：

```powershell
.\scripts\run.ps1 run iot-mqtt-simulator --devices 4 --interval 1 --qos 0
.\scripts\run.ps1 run iot-mqtt-simulator --devices 4 --interval 1 --qos 1
.\scripts\run.ps1 run iot-mqtt-simulator --devices 4 --interval 1 --qos 2
```

建议记录表格：

| QoS | 设备数 | 发送间隔 | 运行时间 | 消息数 | 平均延迟(ms) | 最大延迟(ms) | 现象 |
|---|---:|---:|---:|---:|---:|---:|---|
| 0 | 4 | 1s | 60s | TODO | TODO | TODO | TODO |
| 1 | 4 | 1s | 60s | TODO | TODO | TODO | TODO |
| 2 | 4 | 1s | 60s | TODO | TODO | TODO | TODO |

理论分析：

- QoS 0 没有确认流程，协议开销最低，适合允许少量丢失的周期性传感器数据。
- QoS 1 需要确认，可靠性更高，但可能产生重复消息。
- QoS 2 通过更复杂的确认流程保证只投递一次，可靠性最高，但延迟和协议交互开销通常也更高。

本实验在本机环境运行，因此网络丢包、抖动较少，不同 QoS 的延迟差异可能不明显。若要进一步观察差异，可增加设备数量、提高发送频率，或在网络中人为加入丢包和延迟。

### 5.3 CoAP RTT 实验

实验目的：验证 CoAP 请求/响应模式的数据上报，并统计 RTT。

实验命令：

```powershell
.\scripts\run.ps1 run iot-coap-server
.\scripts\run.ps1 run iot-coap-client --count 5 --interval 0.1
```

一次测试输出示例：

```text
001 2.04 Changed rtt=3.404ms payload={"status": "ok", ...}
002 2.04 Changed rtt=1.089ms payload={"status": "ok", ...}
003 2.04 Changed rtt=0.961ms payload={"status": "ok", ...}
004 2.04 Changed rtt=1.198ms payload={"status": "ok", ...}
005 2.04 Changed rtt=1.025ms payload={"status": "ok", ...}

CoAP RTT summary
count: 5
avg:   1.536 ms
min:   0.961 ms
max:   3.404 ms
```

结果说明 CoAP 服务端能够正确接收传感器数据，并返回 `2.04 Changed` 响应。由于 CoAP 基于 UDP，单次请求/响应流程较轻量，在本机环境下 RTT 较低。

### 5.4 MQTT 与 CoAP 对比

| 对比项 | MQTT | CoAP |
|---|---|---|
| 通信模型 | 发布/订阅 | 请求/响应 |
| 底层传输 | TCP | UDP |
| 是否依赖中间节点 | 依赖 Broker | 通常不依赖 Broker |
| 多设备消息分发 | 适合 | 相对不如 MQTT 自然 |
| 单次状态上报 | 可以 | 适合 |
| 可靠性机制 | QoS 0/1/2 | Confirmable Message、重传 |
| 适用场景 | 云平台订阅、多设备数据汇聚、消息推送 | 受限设备、局域网资源访问、轻量 REST 接口 |

从实验结果看，MQTT 更适合本实验中的“多个传感器持续上报，监控平台统一订阅”的场景。它通过 Broker 解耦了设备和订阅者，扩展多个设备时只需新增 Topic。CoAP 更像轻量版 HTTP，适合设备直接向某个资源上报数据，模型简单，协议开销较小。

## 6. 结果分析

本实验实现了一个完整的 MQTT 数据采集闭环：传感器模拟器持续生成数据，MQTT Broker 负责消息转发，Dashboard 订阅消息后保存到 SQLite 并进行页面展示。实验说明 MQTT 的发布/订阅模型适合物联网多设备数据采集，Topic 可以自然表达设备编号和传感器类型。

从延迟统计看，本机环境下 MQTT 和 CoAP 的 RTT/端到端延迟都较低。需要注意的是，本实验主要验证协议机制和系统功能，不代表真实广域网或低功耗无线网络中的性能。在真实环境中，网络质量、Broker 部署位置、设备性能、QoS 等级和消息频率都会影响最终延迟。

从可靠性角度看，MQTT QoS 提供了清晰的可靠性等级选择。周期性传感器数据可以选择 QoS 0 或 QoS 1，因为下一次上报可以覆盖上一次数据；告警类消息更适合使用 QoS 1 或 QoS 2，以降低漏报风险。CoAP 则适合简单设备直接向服务端发送状态，尤其适用于资源受限设备和局域网场景。

## 7. 不足与改进

本实验仍存在一些不足：

- 实验主要运行在本机环境，缺少真实网络丢包、延迟和抖动条件。
- MQTT QoS 对比目前主要统计延迟，没有统计重复消息和丢失消息。
- Dashboard 当前以表格和统计卡片为主，尚未加入折线图展示历史趋势。
- CoAP 部分实现了简化 POST 上报，没有进一步实现资源发现、Observe 等机制。
- SQLite 适合实验原型，若用于长期运行系统，应考虑时序数据库或消息队列。

后续可以从以下方向改进：

- 引入网络模拟工具，设置不同丢包率和延迟，观察 QoS 差异。
- 增加消息序号，统计丢包率、重复率和乱序情况。
- 增加历史曲线图，展示传感器数据变化趋势。
- 为 CoAP 增加 Confirmable/Non-confirmable 对比实验。
- 将 Broker 和 Dashboard 部署到不同主机，测试跨网络通信效果。
